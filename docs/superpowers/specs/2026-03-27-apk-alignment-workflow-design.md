# 看盘啦：APK 基准驱动的结果提取、逆向与自动化回归设计

日期：2026-03-27

## 1. 背景

当前项目是一个基于 SwiftUI 的 iOS 紫微斗数排盘应用，产品名为 `看盘啦`。项目的本质目标不是独立设计一套新规则，而是尽量复刻既有 Android APK 的排盘结果与行为。

用户已经对当前版本做过比较细致且较全面的人工测试。现阶段剩下的问题不是大块逻辑缺失，而是仍存在一些细小差异。因此，当前最缺的不是继续大规模手工对照，而是建立一套以 APK 输出为唯一基准的结果获取、结构化与自动化回归机制。

## 2. 当前已有能力

### 2.1 产品与功能

- 支持北京时间、真太阳时、阴历时间三种输入模式
- 支持闰月、换月、经度、姓名、事项备注等输入项
- 支持生成紫微盘面与文字详情
- 支持点击宫位查看单宫详情
- 结果页提供 APK 风格的调试参数串，便于复现输入

### 2.2 工程与质量

- 使用 SwiftUI、Swift 5.9、iOS 17+
- 通过 `project.yml` 作为 XcodeGen 单一配置源
- 已有基础单元测试：输入解析、农历转换、真太阳时、引擎纯函数
- 已配置共享 Scheme 与 GitHub Actions 测试流程

### 2.3 已知边界

- 项目已收缩到紫微排盘主线，不再继续八字方向
- APK 中依赖 native 库的 `层2.1` / `层3.1` 未实现，当前不作为首要范围
- 当前仓库里只有 `app.apk` 文件，尚未建立稳定的 APK 结果提取链路

## 3. 核心问题

### 3.1 唯一基准

本项目后续所有验证与逆向工作的唯一基准都是 APK 实际跑出来的结果，而不是当前 iOS 工程自身的输出。

### 3.2 当前缺口

当前缺的不是“是否继续人工测试”，而是以下三件事：

- 稳定拿到 APK 的排盘结果
- 将 APK 结果整理为统一的结构化数据
- 基于这些结构化结果建立自动化回归，并反推规则逻辑

### 3.3 当前起点

- 用户当前手上主要只有 `app.apk`
- 因此必须先解决 APK 的可运行与结果提取问题
- 之后才能进入结果结构化、逻辑逆向与自动化测试

## 4. 本轮目标

### 4.1 第一优先主线

- 建立 APK 结果获取能力
- 建立 APK 结果的结构化模型
- 建立以 APK 结果为基准的自动化边界回归

### 4.2 第二优先主线

- 恢复本机运行 APK 的能力，使结果获取可以持续进行；这是支撑第一优先主线的最小可行手段，而不是独立于结果基准之外的并列目标
- 允许先从 APK 界面可见结果入手，同时并行探索内部数据提取路径

### 4.3 下一阶段重要主线

- UI / 交互优化仍然重要，但不是当前第一优先
- 下一阶段 UI 优化的重点不是纯视觉美化，而是输入流程优化

## 5. 方法论与原则

### 5.1 基本链路

本轮采用如下方法链路：

`APK 输出 -> 结构化结果 -> 反推逻辑 -> 自动化回归`

### 5.2 核心原则

- APK 输出是唯一基准
- 不以当前 iOS 输出作为最终真值
- 先获取结果，再谈逆向逻辑
- 先建立稳定数据形态，再扩展测试覆盖
- 先围绕剩余细小差异建立保护网，而不是重复已有的大量人工验证

## 6. APK 结果获取策略

### 6.1 路径 A：界面读取优先

先让 APK 跑起来，并从盘面、详情页、宫位详情等 UI 中读取结果，形成第一版结构化数据。

优点：

- 最快落地
- 不要求先理解 APK 内部实现
- 适合快速建立第一版结果采集流程

缺点：

- 容易受到 UI 展示方式影响
- 自动化程度有限

### 6.2 路径 B：内部数据提取优先

尝试从 APK 内部更原始的数据来源、处理链路、接口、日志或相关线索中提取结果。

优点：

- 更接近原始基准
- 更适合后续自动化与逻辑逆向

缺点：

- 不确定性更高
- 可能遇到混淆或 native 层阻碍

### 6.3 路径 C：双轨策略

先用界面读取建立可落地的结果采集能力，同时并行探索内部数据提取；一旦内部路径可行，再逐步将其升级为主基准来源。

这里“恢复 APK 运行能力”的定位是结果采集手段，而不是本轮的独立终点。只要某种环境能够稳定复现 APK 输出，就可以作为合法执行路径，不强制限定必须是单一本机方案。

本轮采用 `路径 C`，但执行顺序明确如下：

1. 先恢复 APK 的运行能力
2. 先从界面可见结果建立第一版采集链路
3. 再并行探索内部数据提取
4. 未来如果内部提取更稳定，则切换为主基准源

## 7. 结构化结果模型

### 7.1 输入层

记录盘面的输入条件，确保样本可复现：

- 时间输入模式
- 年月日时分
- 性别
- 经度
- 闰月
- 换月
- APK 原始输入串

### 7.2 全局结果层

记录整张盘的全局结论，第一批重点包括：

- 命宫
- 身宫
- 五行局
- 命主 / 身主
- 来因宫
- 流年干支
- 流斗
- 虚岁
- 四化总表

### 7.3 宫位结果层

记录十二宫逐宫结果，支持后续逆向与回归：

- 宫名
- 地支位置
- 宫干
- 主星 / 辅星 / 杂曜列表
- 大限
- 小限
- 长生
- 岁前 / 将前 / 博士
- 宫干四化 / 自化 / 冲化
- 其他额外标记

### 7.4 来源与可信度层

为每条结果记录来源与确认状态，例如：

- 来自 APK UI 可见文本
- 来自详情页
- 来自人工整理
- 来自内部数据提取
- 是否已经人工确认

结构化模型可以先为后续扩展保留较宽字段范围，但第一批自动化断言只按第 8.3 节启用，不要求首版一次性覆盖和校验全部字段。

## 8. 自动化回归设计

### 8.1 总体方向

自动化回归的目标不是让 iOS 自证正确，而是让 iOS 输出持续对齐 APK 基准。

### 8.2 回归形式

采用以 APK 结果为来源的结构化快照式回归：

- 先采集并保存 APK 基准结果
- 将其转换为统一的数据结构
- 由测试对 iOS 输出与这些结构化基准做自动比对

### 8.3 第一批优先覆盖

第一批自动化边界回归优先覆盖排盘结果规则，而不是 UI 行为：

- 命宫 / 身宫
- 四化
- 大限方向与年龄段
- 主要星曜落宫

### 8.4 当前明确不先做

- 不先做 UI 自动化测试
- 不先做截图快照测试
- 不先做以 iOS 输出为真值的测试基线
- 不先把整张命盘所有字段一次性全部绑死

## 9. UI / 交互优化的位置

### 9.1 位置

UI / 交互优化保留为下一阶段重要主线，但不压过当前的 APK 基准与自动化回归工作。

### 9.2 重点

当前确认的下一阶段重点是输入流程优化，包括：

- 录入信息更快、更顺手
- 模式切换更自然
- 减少重复操作与跳转成本
- 为后续对照工作流提供更顺畅的录入体验

## 10. 非目标

- 不先把重心放在重建完整 Android 开发环境
- 不先建设重型 APK 自动抓取平台
- 不先做完整 native 层逆向
- 不先扩展新的产品功能
- 不先进行大规模 UI 改版
- 不以当前 iOS 输出作为最终基准

## 11. 本轮完成标准

满足以下条件即可认为当前阶段的设计目标达成：

- 已建立至少一种可持续的 APK 结果获取路径
- 已能把至少一批 APK 排盘结果整理为统一结构化数据
- 已定义并落地第一批自动化回归的基准形态
- 自动化回归已优先围绕以下核心结果组织：命宫 / 身宫、四化、大限方向与年龄段、主要星曜落宫
- “反推逻辑”在本阶段以支持结果解释与后续测试设计为目标，不要求本阶段完成完整规则文档化或完整逆向闭环
- 后续 iOS 变更不再只依赖人工经验，而有 APK 基准驱动的自动化保护网作为目标方向

## 12. 后续计划入口

本设计确认后，下一步进入实施计划阶段，输出一份围绕以下主题的执行计划：

- 恢复 APK 运行与结果获取能力
- 建立结构化结果模型
- 建立 APK 基准驱动的自动化回归骨架
- 将“反推逻辑”控制在支持结果解释、差异定位与测试设计的必要范围内
- 为下一阶段的输入流程优化预留接口与工作边界
