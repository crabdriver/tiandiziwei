## 一、项目定位与现状

### 1.1 是什么

- **形态**：基于 SwiftUI 的 **iOS 应用**，产品名 **看盘啦**（工程目录 / target 名仍为 `ZiWeiDoushuDianLiangXingKong`）。
- **核心目标**：紫微斗数排盘逻辑继承「点亮星空」体系，**与既有 Android APK 的实际行为尽量对齐**，便于对照样本做校准与回归。
- **技术栈**：Swift 5.9、最低系统 **iOS 17**、工程配置以 **`project.yml`（XcodeGen）为单一来源**（生成 `project.pbxproj`），流程见 **`docs/工程配置说明.md`**。

### 1.2 当前能力（工程与产品）

| 维度 | 说明 |
|------|------|
| 排盘核心 | `ZiWeiEngine` + 农历/真太阳时等：十二宫、主星/辅星/煞曜/杂曜、四化、大限、流年相关字段等（详见 `README.md`「已对齐」列表） |
| 输入 | 北京时间 / 真太阳时 / 阴历、闰月、换月、经度等；与 APK 参数串互转（`ChartInput`） |
| 展示 | 命盘网格 `ZiWeiChartView`、详情列表 `ChartDisplayView`；盘面 **点击宫位** 弹出与列表一致的 **单宫详情**（`PalaceDetailContent`） |
| 界面 | **暖纸/夜空主题**：全页渐变底；`InputView` 英雄区圆角、渐变描边与阴影；`ChartDisplayView` 毛玻璃分段（图标 + 文案、弹簧动画）、命盘轻阴影；导航栏 `ultraThinMaterial`；宫名 **朱砂色**；`ColorTheme.swift`（含 `ZiWeiColors` / `AppColors` 渐变对齐）与 `ChartPalette` |
| 调试向 | 「APK 对照」默认收起于 **开发者** 折叠区，减少对普通用户的干扰 |
| 工程 | `ChartPalette` 统一星曜/四化展示色；**单元测试**（解析层、`LunarCalendarConverter`、`TrueSolarTime`、`ZiWeiEngine` 纯函数等，**不跑整盘**）；**GitHub Actions** 在 push/PR 上跑模拟器测试；`project.yml` + 共享 Scheme 便于 `xcodegen` 与 CI 共用同一套 Scheme |
| 明确未做 | APK 中 **`层2.1` / `层3.1`** 等依赖 native 分析库的能力，**不属于传统排盘规则**，iOS 侧未实现；模型中相关字段保留作扩展位 |

## 三、想做的事（方向）

按优先级大致如下（可根据你个人精力调整）：

1. **持续对齐 APK**  
   - 新样本、边界 case（节气、子时、闰月、换月等）发现差异时，以现有回归思路改引擎并更新文档/样本记录。

2. **工程质量**  
   - 继续扩展 **单元测试**（解析层、纯函数）；排盘整盘仍以样本对照为主。  
   - **只维护 `project.yml`**，改工程后执行 `xcodegen generate`（见 `docs/工程配置说明.md`）；改 Scheme 时注意与已提交 `.xcscheme` / CI 的 `-scheme`、`-destination` 一致。  
   - **CI 维护**：GitHub Actions 已启用；后续可按需在 workflow 中 **固定 Xcode 版本** 或调整模拟器 **`-destination`**。若使用 **GitLab** 等，须在 **macOS + Xcode** 的 Runner 上跑 `xcodebuild`（Linux 共享 Runner 无法替代）。

3. **产品体验（紫微主线）**  
   - 在「不加八字」前提下，继续打磨 **可读性**（字号、密度）、**暗色模式**（若需要）、**无障碍**、以及「开发者」区块的入口区隔。

4. **可选扩展（与 APK 一致前提下）**  
   - 若未来需要 **层2/层3** 类能力，需单独评估是否引入等价计算或接口，而非混在排盘内核里。
